# 4. Git Avançado

## 4.1 Branches (Ramificações)

Branches permitem que você trabalhe em diferentes linhas de desenvolvimento simultaneamente.

### O que são Branches?

```
main        ●────────●────────●────────●
                   \         \
feature-login      ●────────●───────────●
                         \
feature-pagamento        ●───────────────●
```

### Comandos de Branch:

```bash
# Listar branches locais
git branch

# Listar todas as branches (locais e remotas)
git branch -a

# Criar nova branch (sem mudar para ela)
git branch nome-da-branch

# Criar e mudar para a branch
git checkout -b nome-da-branch

# Método moderno (Git 2.23+)
git switch -c nome-da-branch

# Mudar de branch
git checkout nome-da-branch
# ou
git switch nome-da-branch

# Renomear branch atual
git branch -m novo-nome

# Deletar branch local
git branch -d nome-da-branch

# Deletar branch local forçadamente
git branch -D nome-da-branch

# Deletar branch remota
git push origin --delete nome-da-branch
```

### Branch Naming Conventions:

| Prefixo | Uso | Exemplo |
|---------|-----|---------|
| `feature/` | Novas funcionalidades | `feature/autenticacao` |
| `bugfix/` | Correção de bugs | `bugfix/erro-calculo` |
| `hotfix/` | Correções urgentes | `hotfix/vazamento-dados` |
| `docs/` | Documentação | `docs/atualiza-readme` |
| `refactor/` | Refatoração | `refactor/modulo-pagamento` |
| `test/` | Testes | `test/adiciona-tests-unitarios` |

## 4.2 Merge (Mesclagem)

Unir o histórico de duas ou mais branches.

### Tipos de Merge:

#### 1. Fast-Forward Merge
Ocorre quando a branch alvo não teve novas mudanças.

```
main      ●────────●
               \
feature       ●─●─●

Após merge (fast-forward):
main      ●────────●─●─●
```

```bash
git checkout main
git merge feature
```

#### 2. Three-Way Merge (Merge Commit)
Ocorre quando ambas as branches tiveram mudanças.

```
main      ●────────●───────● (merge commit)
               \         /
feature       ●─●─●─────●
```

```bash
git checkout main
git merge --no-ff feature  # Força merge commit
```

#### 3. Squash Merge
Combina todos os commits da feature em um único commit.

```bash
git merge --squash feature
git commit -m "Adiciona feature completa"
```

### Resolvendo Conflitos de Merge:

Quando o Git não consegue mesclar automaticamente:

```bash
# 1. Tente o merge
git merge feature

# Saída: CONFLICT (content): Merge conflict in arquivo.txt

# 2. Abra o arquivo com conflito
# Você verá marcadores assim:
<<<<<<< HEAD
Código da branch atual
=======
Código da branch que está sendo mesclada
>>>>>>> feature

# 3. Edite o arquivo, mantendo o código desejado
# 4. Adicione o arquivo resolvido
git add arquivo.txt

# 5. Complete o merge
git commit
```

**Dica:** Use uma ferramenta visual para resolver conflitos:
```bash
git mergetool  # Abre ferramenta configurada (meld, kdiff3, etc.)
```

## 4.3 Rebase

Rebase reescreve o histórico, movendo uma branch para começar em um commit diferente.

### Rebase vs Merge:

```
Antes do rebase:
main      ●────────●────────●
               \
feature       ●─●─●

Após rebase:
main      ●────────●────────●
                          \
feature                   ●─●─● (rebaseado)
```

### Comandos de Rebase:

```bash
# Rebase simples
git checkout feature
git rebase main

# Rebase interativo (permite editar commits)
git rebase -i HEAD~3

# Rebase com squash (combina commits)
git rebase -i --squash HEAD~3

# Abortar rebase em caso de problemas
git rebase --abort

# Continuar rebase após resolver conflitos
git rebase --continue
```

### Rebase Interativo:

```bash
git rebase -i HEAD~4
```

Editor abrirá com opções:

```
pick abc123 Primeiro commit
pick def456 Segundo commit
pick ghi789 Terceiro commit
pick jkl012 Quarto commit

# Comandos disponíveis:
# p, pick <commit> = usar commit
# r, reword <commit> = usar commit, editar mensagem
# e, edit <commit> = usar commit, parar para amend
# s, squash <commit> = usar commit, mesclar com anterior
# f, fixup <commit> = como squash, mas descarta mensagem
# d, drop <commit> = remover commit
```

### ⚠️ Quando NÃO usar Rebase:

> **NUNCA** faça rebase em branches públicas/compartilhadas! Isso reescreve o histórico e causa problemas para outros desenvolvedores.

**Use rebase apenas em:**
- ✅ Branches locais/private
- ✅ Antes de merge para limpar o histórico

## 4.4 Tags

Tags são marcadores permanentes para commits específicos, geralmente usados para versões.

### Tipos de Tags:

#### Tag Leve (Lightweight):
```bash
git tag v1.0.0
```

#### Tag Anotada (Annotated) - Recomendado:
```bash
git tag -a v1.0.0 -m "Versão 1.0.0 - Lançamento inicial"
```

### Comandos de Tag:

```bash
# Listar tags
git tag

# Listar tags que combinam com padrão
git tag -l "v1.*"

# Mostrar informações de uma tag
git show v1.0.0

# Criar tag em commit específico
git tag -a v1.0.0 abc123 -m "Versão 1.0.0"

# Enviar tag para remoto
git push origin v1.0.0

# Enviar todas as tags
git push origin --tags

# Deletar tag local
git tag -d v1.0.0

# Deletar tag remota
git push origin --delete v1.0.0
```

### Versionamento Semântico (SemVer):

```
MAIOR.MENOR.CORREÇÃO
  │     │      │
  │     │      └─ Correções de bugs (backward compatible)
  │     └─ Novas funcionalidades (backward compatible)
  └─ Mudanças incompatíveis

Exemplos:
1.0.0 → Primeiro lançamento estável
1.0.1 → Correção de bug
1.1.0 → Nova funcionalidade
2.0.0 → Mudança incompatível
```

## 4.5 Stash

Stash guarda temporariamente alterações não commitadas.

### Quando Usar:

> Você está trabalhando em uma feature, precisa mudar de branch urgentemente, mas não quer commitar código incompleto.

### Comandos de Stash:

```bash
# Guardar alterações
git stash

# Guardar com mensagem descritiva
git stash save "Trabalhando na feature X"

# Ou método moderno
git stash push -m "Trabalhando na feature X"

# Listar stashes
git stash list

# Aplicar último stash (mantém na lista)
git stash apply

# Aplicar stash específico
git stash apply stash@{2}

# Aplicar e remover da lista
git stash pop

# Remover stash específico
git stash drop stash@{1}

# Limpar todos os stashes
git stash clear

# Ver conteúdo de um stash
git stash show stash@{0}

# Ver conteúdo completo
git stash show -p stash@{0}

# Criar branch a partir de stash
git stash branch nova-branch stash@{0}
```

### Exemplo de Uso:

```bash
# 1. Você está trabalhando em uma feature
git checkout -b feature/nova
# Faz alterações...

# 2. Precisa mudar urgentemente para main
git stash save "Feature nova - em andamento"

# 3. Muda para main e resolve o problema
git checkout main
# ...correção...
git commit -m "Hotfix importante"

# 4. Volta para feature e restaura alterações
git checkout feature/nova
git stash pop
```

## 4.6 Cherry-Pick

Copia um commit específico de uma branch para outra.

### Quando Usar:

- Aplicar um hotfix de uma branch para outra
- Recuperar um commit que foi perdido
- Selecionar commits específicos de uma PR

### Comandos:

```bash
# Copiar commit para branch atual
git cherry-pick abc123

# Copiar múltiplos commits
git cherry-pick abc123 def456 ghi789

# Copiar intervalo de commits
git cherry-pick abc123..def456

# Cherry-pick sem commitar automaticamente
git cherry-pick --no-commit abc123
```

### Exemplo:

```bash
# Hotfix foi feito na branch main
git log main --oneline
# abc123 (HEAD) Hotfix: corrige vulnerabilidade

# Aplicar hotfix na branch de release
git checkout release/v1.0
git cherry-pick abc123
```

## 4.7 Reset, Restore e Revert

### git reset (Move HEAD e desfaz commits)

```bash
# Desfaz commits, mantém alterações no working directory
git reset --soft HEAD~1

# Desfaz commits, mantém alterações staged
git reset --mixed HEAD~1  # (padrão)

# Desfaz commits, descarta todas as alterações
git reset --hard HEAD~1

# ⚠️ CUIDADO: --hard descarta alterações permanentemente!
```

### git restore (Desfaz alterações em arquivos)

```bash
# Descartar alterações de um arquivo (não staged)
git restore arquivo.txt

# Descartar alterações staged
git restore --staged arquivo.txt

# Descartar tudo (staged e working directory)
git restore --staged --worktree arquivo.txt
```

### git revert (Cria novo commit que desfaz outro)

```bash
# Reverter último commit
git revert HEAD

# Reverter commit específico
git revert abc123

# Reverter sem abrir editor
git revert --no-edit abc123

# Reverter intervalo de commits
git revert abc123..def456
```

### Comparação:

| Comando | Uso | Histórico |
|---------|-----|-----------|
| `reset` | Desfaz commits locais | Reescreve histórico |
| `restore` | Descarta alterações em arquivos | Não altera histórico |
| `revert` | Desfaz commit com novo commit | Preserva histórico |

> ✅ **Use `revert`** em branches públicas (não reescreve histórico)
> ⚠️ **Use `reset`** apenas em branches locais

## 4.8 Reflog

Reflog registra todos os movimentos do HEAD. É seu "seguro" contra acidentes!

### Comandos:

```bash
# Ver histórico do HEAD
git reflog

# Ver reflog de uma branch
git reflog show main

# Voltar para estado anterior
git reset --hard HEAD@{5}

# Recuperar commit "perdido"
git checkout -b branch-recuperada abc123
```

### Cenário de Recuperação:

```bash
# 😱 Ops! Deletei a branch por acidente!
git branch -D feature-importante

# 😰 Como recupero?
git reflog
# Saída:
# abc123 HEAD@{0}: checkout: moving to main
# def456 HEAD@{1}: commit: Feature importante

# ✅ Recupera!
git checkout -b feature-recuperada def456
```

## 4.9 Hooks do Git

Hooks são scripts que rodam automaticamente em eventos do Git.

### Hooks Disponíveis:

| Hook | Quando roda |
|------|-------------|
| `pre-commit` | Antes do commit |
| `commit-msg` | Após editar mensagem do commit |
| `pre-push` | Antes do push |
| `post-merge` | Após merge |
| `pre-receive` | No servidor, antes de push |

### Exemplo: Hook pre-commit

```bash
# .git/hooks/pre-commit
#!/bin/bash

# Rodar testes antes de commitar
npm test

# Verificar formatação do código
npm run lint

# Se algum comando falhar, o commit é abortado
```

### Hooks com Husky (Recomendado):

```bash
# Instalar Husky
npm install husky --save-dev
npx husky install

# Adicionar hook
npx husky add .husky/pre-commit "npm test"
```

## 4.10 Boas Práticas

### Mensagens de Commit:

```
❌ Ruim:
- "arrumei"
- "teste"
- "update"
- "fix"

✅ Bom:
- "Corrige cálculo de imposto no módulo financeiro"
- "Adiciona validação de CPF no formulário"
- "Remove dependência obsoleta do package.json"
```

### Estrutura de Commit Convencional:

```
<tipo>(<escopo>): <descrição>

[corpo opcional]

[rodapé opcional]
```

**Tipos:**
- `feat:` Nova funcionalidade
- `fix:` Correção de bug
- `docs:` Documentação
- `style:` Formatação/código estético
- `refactor:` Refatoração
- `test:` Testes
- `chore:` Tarefas de build/config

**Exemplo:**
```
feat(auth): adiciona login com Google

Implementa autenticação via OAuth2 do Google.
Adiciona botão "Login com Google" na tela de login.

Closes #123
```

### Fluxo de Trabalho Recomendado:

```bash
# 1. Sempre comece da main atualizada
git checkout main
git pull origin main

# 2. Crie branch para feature
git checkout -b feature/nova-funcionalidade

# 3. Trabalhe e faça commits atômicos
git add .
git commit -m "feat: adiciona componente X"

# 4. Antes de push, rebase na main
git checkout main
git pull origin main
git checkout feature/nova-funcionalidade
git rebase main

# 5. Push e crie Pull Request
git push -u origin feature/nova-funcionalidade
```

### Comandos de Segurança:

```bash
# Sempre verifique antes de push
git status
git log --oneline -5
git diff main

# Use dry-run para ver o que será pushado
git push --dry-run
```
